<p><b><font face="helvetica" size="12">Discussion:</font></b></p>
<p>The <font face="Courier" size="11">keyDown</font><font face="helvetica" size="12"> message is sent whenever a character key is pressed. If the blinking text cursor is in a field, the </font><font face="Courier" size="11">keyDown</font><font face="helvetica" size="12"> message is sent to that field. When the </font><font face="Courier" size="11">keyDown</font><font face="helvetica" size="12"> message is sent to the field, it triggers this handler.</font></p>
<p></p>
<p>The Revolution engine must receive the <font face="Courier" size="11">keyDown</font><font face="helvetica" size="12"> message in order to respond to a typed character (or in other words, the entry of a character is triggered by the </font><font face="Courier" size="11">keyDown</font><font face="helvetica" size="12"> message), so trapping the message and not passing it prevents typing from being entered in the field.</font></p>
<p></p>
<p>When the user types a key, the <font face="Courier" size="11">keyDown</font><font face="helvetica" size="12"> handler checks to see whether the field already contains the maximum number of characters allowed. The keyword </font><font face="Courier" size="11">me</font><font face="helvetica" size="12"> refers to the object whose script contains the handler that is executing. In this case, </font><font face="Courier" size="11">me</font><font face="helvetica" size="12"> refers to the field, because the </font><font face="Courier" size="11">keyDown</font><font face="helvetica" size="12"> handler is in the field's script. When a field is referenced in this way, Revolution returns the contents of the field. The length of that content is the number of characters it contains. So the expression </font><font face="Courier" size="11">the length of me</font><font face="helvetica" size="12"> is the number of characters in the field.</font></p>
<p></p>
<p>If the field already contains the maximum number of characters, the handler beeps and doesn't <font face="Courier" size="11">pass</font><font face="helvetica" size="12"> the keyDown message on, which causes the key press to be </font><i><font face="Lucida Grande" size="11">lost</font></i><font face="helvetica" size="12">. If the field does not yet contain the maximum number of characters, the handler uses the </font><font face="Courier" size="11">pass</font><font face="helvetica" size="12"> control structure to allow the </font><font face="Courier" size="11">keyDown</font><font face="helvetica" size="12"> message to proceed along the message path, which causes the character to be added to the field normally.  For more details on the message path, see the section The </font><i><font face="Lucida Grande" size="11">Message Path</font></i><font face="helvetica" size="12"> in the </font><i><font face="Lucida Grande" size="11">User's Guide</font></i><font face="helvetica" size="12">.</font></p>
<p></p>
<p><b><font face="Lucida Grande" size="11">Note:</font></b><font face="helvetica" size="12">  The </font><font face="Courier" size="11">keyDown</font><font face="helvetica" size="12"> message is sent for character keys, but not for non-character keys such as the arrow keys. If you want to trap such keys, or trap paste events, you'll need to use a </font><font face="Courier" size="11">rawKeyDown</font><font face="helvetica" size="12"> handler instead. However, for simple data entry fields, a </font><font face="Courier" size="11">rawKeyDown</font><font face="helvetica" size="12"> handler is a little more complicated to implement and is usually overkill.</font></p>
