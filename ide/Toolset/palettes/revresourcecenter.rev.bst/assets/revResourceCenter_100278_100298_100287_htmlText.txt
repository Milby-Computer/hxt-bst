<p><b><font face="helvetica" size="12">Discussion:</font></b></p>
<p>The <font face="Courier" size="11">mouseDown</font><font face="helvetica" size="12"> message is sent when the user clicks an object. If a menu was clicked, the message is sent before the menu is displayed, so if you make changes to the button in the </font><font face="Courier" size="11">mouseDown</font><font face="helvetica" size="12"> handler, the changes appear in the menu. Remember that in Revolution, menus are implemented as buttons, and a menu bar is a group of menu buttons. When the </font><font face="Courier" size="11">mouseDown</font><font face="helvetica" size="12"> message is sent to the menu button, it triggers this handler.</font></p>
<p></p>
<p><font color="#404040"><img src="202652" char=" "> </font><b><font face="Lucida Grande" size="11">Cross-platform note:</font></b><font face="helvetica" size="12"> When a menu button is being displayed in the Mac OS menu bar, it does not receive </font><font face="Courier" size="11">mouseDown</font><font face="helvetica" size="12"> messages, but its group does. For this reason, this example handler should be placed in the script of the menu bar group, rather than in the menu button. The first line of the handler makes sure it's only executed if the user clicked the Window menu. This ensures that the example will work on all platforms.</font></p>
<p></p>
<p>This <font face="Courier" size="11">mouseDown</font><font face="helvetica" size="12"> handler builds a list of open stack windows whenever the user clicks the menu. Recreating the list every time the menu is clicked ensures that the menu is always current. Any menu that is constructed dynamically must be created in a </font><font face="Courier" size="11">mouseDown</font><font face="helvetica" size="12"> handler, since that is the only way to ensure that the button's contents are updated before the menu is displayed. </font></p>
<p></p>
<p>The handler begins with the <font face="Courier" size="11">openStacks</font><font face="helvetica" size="12"> function, which returns a list of the </font><font face="Courier" size="11">short name</font><font face="helvetica" size="12"> of each stack whose window is open.</font></p>
<p></p>
<p>The handler could put this list into the menu just as it is. However, this would list the name property of the stacks, rather than the <font face="Courier" size="11">title</font><font face="helvetica" size="12"> property. The </font><font face="Courier" size="11">title</font><font face="helvetica" size="12"> is the text shown in a stack's title bar, and is often more descriptive than the stack's </font><font face="Courier" size="11">name</font><font face="helvetica" size="12">. It's also the name the user sees on the screen, so for most situations, it's better to show the user the </font><font face="Courier" size="11">title</font><font face="helvetica" size="12"> instead of the </font><font face="Courier" size="11">name</font><font face="helvetica" size="12">.</font></p>
<p></p>
<p>To get the stack titles, the handler walks through the list of stack names in a <font face="Courier" size="11">repeat</font><font face="helvetica" size="12"> loop, using the </font><font face="Courier" size="11">repeat for each</font><font face="helvetica" size="12"> form. This form of </font><font face="Courier" size="11">repeat</font><font face="helvetica" size="12"> creates a variable (in this handler, the variable is called "thisStack") and places each stack name in turn into the variable. The loop repeats once for each open stack, gathering the </font><font face="Courier" size="11">title</font><font face="helvetica" size="12"> of each stack into a variable called "windowsList" that will be used as the menu's contents.</font></p>
<p></p>
<p>In most cases, a standard Window menu does not include the names of palettes and dialog boxes. To filter out such windows, the handler checks the <font face="Courier" size="11">mode</font><font face="helvetica" size="12"> property of each of the open stacks, and skips over that stack if its mode is greater than 2. The </font><font face="Courier" size="11">mode</font><font face="helvetica" size="12"> property describes what kind of window a stack is in: 1 is an unlocked stack in an editable window, 2 is a locked stack in an editable window, and higher numbers are palettes, dialog boxes, or other ways of displaying a stack. So a </font><font face="Courier" size="11">mode</font><font face="helvetica" size="12"> of 1 or 2 indicates that the stack is in an editable window: the kind of window we want to show in the menu. A stack with a mode of zero is closed but loaded into memory. However, we don't need to worry about this sort of stack, because closed but loaded stacks are not returned in the openStacks function to begin with.  For more information on stack modes, see </font><i><font face="Lucida Grande" size="11">Scripting a User Interface</font></i><font face="helvetica" size="12"> in the </font><i><font face="Lucida Grande" size="11">User's Guide</font></i><font face="helvetica" size="12">.</font></p>
<p></p>
<p>As an extra, if the Alt key (Unix or Windows) or Option key (Mac OS or OS X) is held down, this step is skipped and all open stacks are shown, whatever their mode.</p>
<p></p>
<p>The handler now stores the stack's <font face="Courier" size="11">name</font><font face="helvetica" size="12"> in a variable called "stacksList". This variable is used to collect all the stack names, in the same order as the stack titles. Later on, we'll store this list in a custom property of the menu. The reason for doing this is so that in another handler, we can take a menu choice and get the corresponding stack name. If we want to do anything with a stack chosen from the menu, the stack </font><font face="Courier" size="11">name</font><font face="helvetica" size="12"> (rather than the </font><font face="Courier" size="11">title</font><font face="helvetica" size="12">) is needed, because two open stacks can have the same title. Without the stored list of names, we wouldn't necessarily be able to tell which stack was being referred to.</font></p>
<p></p>
<p>Next, the handler stores the <font face="Courier" size="11">title</font><font face="helvetica" size="12"> of the stack in the variable "theTitle", and then checks to see if the number of words in the </font><font face="Courier" size="11">title</font><font face="helvetica" size="12"> is greater than zero. It checks for the number of words--rather than just checking whether the </font><font face="Courier" size="11">title</font><font face="helvetica" size="12"> is empty--because Revolution uses a one-space </font><font face="Courier" size="11">title</font><font face="helvetica" size="12"> to indicate that a stack should have no title at all in its title bar. Because of this, some stacks may have a </font><font face="Courier" size="11">title</font><font face="helvetica" size="12"> that consists of a single space, instead of being empty, and since showing a space in a menu isn't very helpful, we want to show such a stack's name instead of its </font><font face="Courier" size="11">title</font><font face="helvetica" size="12">. If the </font><font face="Courier" size="11">title</font><font face="helvetica" size="12"> contains at least one word, the handler adds it to a variable called "theMenuContents", which will become the contents of the Window menu. If the </font><font face="Courier" size="11">title</font><font face="helvetica" size="12"> has no words, the handler adds the stack's name, instead of the </font><font face="Courier" size="11">title</font><font face="helvetica" size="12">.</font></p>
<p></p>
<p>When the <font face="Courier" size="11">repeat</font><font face="helvetica" size="12"> loop is done, the variable "theMenuContents" contains a list of all the open windows, in front-to-back order. The windows list is in this order because that's the order the openStacks function returns them in, and our list is based on the value returned by the </font><font face="Courier" size="11">openStacks</font><font face="helvetica" size="12">. If we wanted to use another order, such as alphabetical, we would use the </font><font face="Courier" size="11">sort</font><font face="helvetica" size="12"> command at this point to sort the list:</font></p>
<p></p>
<p><font face="Courier" size="11">  </font><font face="Courier" size="11" color="#0000FF">sort</font><font face="Courier" size="11"> lines of theMenuContents</font></p>
<p></p>
<p>But what if there are no open windows (that aren't palettes or dialog boxes)? In this case, the variable will be empty--but we don't want the Window menu to be empty, so in this case we put "(No windows open" into the variable. The leading parenthesis makes this a disabled menu item, and the menu will display this grayed-out explanation if there are no windows to list.</p>
<p></p>
<p>Finally, the handler puts the variable "theMenuContents" into the Window button. Because this handler is in the menu bar group's script, it can refer to the group using the <font face="Courier" size="11">me</font><font face="helvetica" size="12"> keyword. The expression button "Window" of </font><font face="Courier" size="11">me</font><font face="helvetica" size="12"> refers to the Window button that's part of the menu bar group. The menu contents is now set to the list of stack titles:  when the menu is displayed a moment later, it will show this list.</font></p>
<p></p>
<p><b><font face="Lucida Grande" size="11">Note:</font></b><font face="helvetica" size="12"> Putting text into a button is the same as setting the text property of the button. These two statements are equivalent:</font></p>
<p><font face="Courier" size="11">  </font><font face="Courier" size="11" color="#0000FF">set</font><font face="Courier" size="11"> the </font><font face="Courier" size="11" color="#FF0000">text</font><font face="Courier" size="11"> of button "Window" to theMenuContents</font></p>
<p><font face="Courier" size="11">  </font><font face="Courier" size="11" color="#0000FF">put</font><font face="Courier" size="11"> theMenuContents into button "Window"</font></p>
<p></p>
<p>This <font face="Courier" size="11">mouseDown</font><font face="helvetica" size="12"> handler only takes care of updating the menu for display; it doesn't do anything when the user chooses a menu item. Typically, a Window menu's script includes a </font><font face="Courier" size="11">menuPick</font><font face="helvetica" size="12"> handler that brings the chosen stack to the front (or does whatever other action is appropriate).</font></p>
<p></p>
<p>Dynamic menus--whose contents may change depending on the context--are often used in applications. For example, the wording of the <i><font face="Lucida Grande" size="11">Cut</font></i><font face="helvetica" size="12"> and </font><i><font face="Lucida Grande" size="11">Copy</font></i><font face="helvetica" size="12"> menu items may change depending on what's currently selected, or certain menu items may be enabled or disabled depending on which window is active. You can use the basic technique demonstrated in this example to create other changes in menus as needed.</font></p>
